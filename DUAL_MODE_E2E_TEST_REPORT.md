## 🎯 测试目标

### 功能完整性验证
1. **模式切换** - 验证用户可以在简单模式和电商模式之间自由切换
2. **数据迁移** - 验证升级时Product数据正确迁移到SPU-SKU结构
3. **数据安全** - 验证降级时有明确的警告和二次确认
4. **错误处理** - 验证各种异常情况的处理

### 用户体验验证
1. **界面友好** - 清晰的标题、描述和按钮文案
2. **操作引导** - Tooltip提示、升级说明、注意事项
3. **状态反馈** - 加载进度、成功/失败提示
4. **防止误操作** - 降级需要输入"CONFIRM"

### 技术质量验证
1. **响应式设计** - 在不同屏幕尺寸下正常工作
2. **加载状态** - 异步操作期间正确显示加载状态
3. **按钮禁用** - 防止重复提交和误操作
4. **内存泄漏** - 组件卸载时正确清理资源

## 🔍 关键测试场景

### 场景1: 完整升级流程
```typescript
1. 进入商品管理页面（简单模式）
2. 点击🚀升级按钮
3. 查看升级说明和统计数据
4. 点击"确认升级"
5. 观察加载进度条
6. 验证模式切换到电商模式
7. 检查迁移结果统计
8. 验证返回按钮出现
```

### 场景2: 完整降级流程
```typescript
1. 在电商模式下点击⬅️返回按钮
2. 查看警告信息和风险提示
3. 尝试输入错误内容（按钮保持禁用）
4. 输入"CONFIRM"（按钮启用）
5. 点击"确认降级"
6. 观察加载状态
7. 验证模式切换回简单模式
8. 检查成功提示
```

### 场景3: 边界情况处理
```typescript
1. 无商品时升级（应该成功但迁移0个）
2. 升级过程中尝试关闭对话框（应该阻止）
3. 降级过程中尝试关闭对话框（应该阻止）
4. 迁移结果提示的关闭功能
```

## ✨ 最佳实践

### 测试编写
- ✅ 使用describe嵌套组织测试结构
- ✅ beforeEach中设置统一的初始状态
- ✅ 每个测试独立，不依赖其他测试的结果
- ✅ 使用有意义的测试名称，清晰描述测试意图

### 选择器策略
- ✅ 优先使用role-based选择器（更稳定）
- ✅ 其次使用text内容选择器（易读）
- ✅ 避免使用CSS类名选择器（易变）
- ✅ 为关键元素添加data-testid属性

### 断言策略
- ✅ 使用expect().toBeVisible()而不是expect().toBeTruthy()
- ✅ 设置合理的timeout参数
- ✅ 对异步操作使用await
- ✅ 验证多个相关条件，确保功能完整
