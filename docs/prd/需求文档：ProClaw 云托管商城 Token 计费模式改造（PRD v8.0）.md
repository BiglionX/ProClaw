## 1. 背景与动机

### 1.1 现状

ProClaw 云托管商城当前采用**固定月费 SaaS 套餐模式**，按功能/资源上限分档：

| 套餐 | 月费 | 商品数 | 月订单 | 域名 | AI主题 | 同步 |
|------|------|--------|--------|------|--------|------|
| 免费版 | ¥0 | 20 个 | 10 单 | 子域名 | ❌ | 手动 |
| 基础版 | ¥29 | 200 个 | 100 单 | 自定义 | ✅ | 实时 |
| 专业版 | ¥99 | 2000 个 | 2000 单 | 自定义 | ✅ | 实时 |
| 企业版 | ¥299 | 不限 | 不限 | 自定义 | ✅ | 实时 |

**免费套餐占用了大量服务器资源却无收入，付费套餐价格较高、用户决策门槛高，且用量不均衡的用户付费不公平。**

### 1.2 已有 Token 基础设施

项目实际上已具备相当完善的 Token 系统基础：

| Token 购买 UI | 前端 | ❌ 缺失 |

### 1.3 改造成 Token 模式的动机

1. **降低付费门槛** — 用户可按需小额充值（最低 ¥10 起），无需每月 ¥29 起步
2. **公平计费** — 按实际用量付费，轻量用户省钱，重度用户按比例多付
3. **提高转化率** — 从"每月固定订阅"到"按需充值消费"，用户心理负担更小
4. **统一计费体系** — 与 AI Token 消耗体系合并，形成统一的 Token 经济闭环
5. **收入可预测** — Token 预充值模式现金流更健康

## 2. 目标

### 2.1 产品目标

将云托管商城的**固定月费套餐模式**全面改造为**Token 按量计费模式**，建立统一的 Token 经济体系：

- 所有云商城功能消耗统一 Token（区别于 LLM API Token）
- 保留免费入门体验（新用户赠送体验 Token）
- Token 可通过充值套餐包购买
- 支持余额不足提醒和自动停服保护

### 2.2 设计原则

- **渐进迁移** — 现有付费用户平滑过渡，不中断服务
- **透明计费** — 每项操作消耗 Token 数量明确告知
- **一币通用** — 同一 Token 用于云商城所有计费操作（商品同步、AI主题生成、订单处理、API调用等）
- **兜底保护** — 余额不足时先预警后限流，不直接停服

## 3. 定价策略与经济可持续性分析

> **核心命题**：Token 定价必须在"用户觉得便宜"和"项目有足够利润可持续发展"之间找到平衡。定价过低 → 亏本运营 → 项目死亡；定价过高 → 用户流失 → 同样死亡。

### 3.1 成本结构分析

云托管商城每个活跃用户的月度基础设施成本：

| 成本项 | 单价 | 中等用户用量 | 月成本 | 说明 |
|--------|------|-------------|--------|------|
| PostgreSQL 数据库 | — | 共享实例分摊 | ¥1~2 | Supabase 免费额度 500MB，超量按量计费 |
| CDN 带宽 + 存储 | ¥0.15~2.50/GB | 约 5~10GB | ¥1.5~3 | 商品图片访问流量、页面静态资源 |
| Next.js 计算 (Vercel) | — | ~50K 请求/月 | ¥1~2 | SSR 渲染 + API 路由调用 |
| WebSocket 实时连接 | — | 1 个持久连接 | ¥3~5 | 桌面端实时同步保活通道（最贵单项） |
| SSL 证书 | Let's Encrypt | — | ¥0 | 免费自动签发 |
| DNS 管理 | — | 1 自定义域名 | ¥0.2 | 域名解析管理开销 |
| 监控/日志/告警 | — | 基础监控 | ¥1~2 | 错误追踪、性能监控、用量统计 |
| **合计** | | | **¥8~14/月** | 中等活跃用户 |

> 轻量用户（<50 商品、手动同步）：月成本约 ¥3~5  
> 重量用户（>1000 商品、实时同步、高流量）：月成本可达 ¥20~40

### 3.2 目标毛利率

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 综合毛利率 | **60%~75%** | 对标成熟 SaaS 产品（Shopify 55%、Wix 65%） |
| 初期毛利率 | **40%~50%** | 用户量小时分摊固定成本较高，可接受 |
| 规模毛利率 | **70%+** | 用户量 >1000 后边际成本递减 |

基于成本 ¥8~14/月，目标收入约 **¥20~56/月/活跃用户**，与旧套餐 ¥29~99 基本对齐但按量付费更灵活。

### 3.3 平台 Token 定价

> **1 PT = ¥0.001（即 1,000 PT = ¥1）**  
> 此定价基于成本分析得出，确保每位活跃用户贡献 ¥20~60/月收入，实现 60%~75% 毛利率。

## 4. 核心改造：计费模型

### 4.1 Token 消耗定价表

> 采用"平台 Token"（PT）作为内部计费单位，与 LLM Token 区分。  
> **1 PT = ¥0.001（即 1,000 PT = ¥1）**

| 操作类型 | 消耗 PT | 说明 |
|---------|---------|------|
| **商品同步** | 50 PT/个 | 单个商品从桌面端同步到云端（含首次+更新） |
| **AI 主题生成** | 5000 PT/次 | AI 生成商城主题和样式 |
| **AI 主题微调** | 1000 PT/次 | 对已有主题的单项调整 |
| **订单处理** | 10 PT/单 | 每笔商城订单的创建和处理 |
| **API 写操作** | 5 PT/次 | 外部 API 调用（创建商品、更新订单等写操作） |
| **API 读操作** | 1 PT/次 | 外部 API 调用（查询商品、订单等读操作） |
| **自定义域名** | 2000 PT/月 | 绑定自定义域名的月租 |
| **SSL 证书** | 1000 PT/月 | 自动 SSL 证书管理（企业版免费附带） |
| **静态资源托管** | 1 PT/MB/月 | 商品图片、页面资源的 CDN 托管 |
| **实时同步保活** | 500 PT/天 | 桌面端实时同步通道保持 |
| **客服入口配置** | 1000 PT/月 | 在线客服挂件 |
| **SEO 优化报告** | 3000 PT/次 | AI 生成 SEO 优化建议报告 |
| **数据导出** | 100 PT/次 | 导出商品/订单为 CSV/Excel |

##### 存量月租类目（按月底存量自动扣除）

| 计费项 | 消耗 PT | 计费方式 | 说明 |
|--------|---------|---------|------|
| **商品数据托管** | 2 PT/个/月 | 按月底商品 SKU 数量扣费 | 含商品信息、规格、属性的数据库存储 |
| **图片/CDN 存储** | 1 PT/MB/月 | 按月底实际占用空间扣费 | 商品主图+轮播图的存储和 CDN 分发 |
| **订单数据留存** | 1 PT/百单/月 | 超 1000 单部分计费 | 订单历史数据归档存储，前 1000 单免费 |
| **商城页面托管** | 500 PT/月 | 固定月费 | 商城首页+商品页+购物车+结算页的基础托管 |

> **设计意图**：纯 Token 操作计费有一个天然盲区——操作收费了，但存下来的数据持续占用存储和 CDN 资源而无对应收入。存量月租弥补了这个缺口，确保"囤积型用户"也合理付费。同时单价极低（200 个商品月租 ≈ ¥0.40），轻量用户几乎无感。
>
> **不设硬性存储上限**：Token 模式下无需"200 个商品上限"这类硬限制，用户想存多少存多少，月底按实际存量扣 Token。余额不足则进入欠费保护流程。

### 4.2 免费额度（新用户赠送）

| 赠送项 | 数量 | 说明 |
|--------|------|------|
| 注册赠送 PT | 50,000 PT（≈¥50） | 新用户完成注册即赠送 |
| 商品同步 | 前 20 个商品免费同步 | 体验商城基础功能 |
| 订单处理 | 前 10 单免费 | 体验订单流转 |
| 子域名 | 永久免费 | `xxx.proclaw.cc` 形式 |
| 商品托管 | 20 个商品永久免月租 | 与免费同步配套 |
| 图片存储 | 100 MB 永久免月租 | 覆盖基础商品图片需求 |
| 商城页面托管 | 永久免费 | 基础页面托管不收费 |

> 免费额度相当于当前"免费版"的能力范围，足够用户完整体验核心流程。存量免费用完前不产生任何费用。

### 6.2 需新增字段/表

ADD COLUMN IF NOT EXISTS resource_type TEXT 
CHECK(resource_type IN ('sync', 'order', 'theme', 'domain', 'cdn', 'api', 'storage', 'hosting', 'retention', 'other'));

-- 用户 Token 配置表：存储用户的个性化 Token 设置

    user_id UUID REFERENCES profiles(id) ON DELETE CASCADE PRIMARY KEY,

    sort_order INTEGER DEFAULT 0,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    UNIQUE(resource_type)
);

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| **现有付费用户流失** | 高 | 平滑迁移+等值折算 PT + 首月额外赠送 20% PT |
| **支付集成周期长** | 中 | Phase 1 可先用"模拟支付"上线，真实支付作为 Phase 2 |
| **Token 定价不合理** | 中 | 初期采用建议定价，Admin 后台支持动态调整 `token_pricing_rules` |
| **用户不理解 Token 模式** | 低 | 定价页增加费用估算计算器，直观展示 |
| **欠费导致服务中断投诉** | 中 | 3天宽限期 + 余额不足多次提醒 + 只停写不停读 |
