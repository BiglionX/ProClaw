### 1. 背景与目标

#### 1.1 背景
- ProClaw 已有完整的角色权限体系（老板、财务、采购、仓库、销售）。
- 当老板邀请新员工加入系统时，需要为新员工分配角色，并确保员工安装 App 后看到的菜单和数据权限与角色匹配。
- 当前流程：老板需手动在桌面端创建员工账号并分配角色，再将下载链接和账号信息告知员工。员工安装后需手动登录，但账号已存在，权限已设定。

#### 1.2 目标
- 简化老板邀请员工的流程：一键生成邀请链接/二维码，员工扫码安装 App 后，自动完成账号创建（或关联已有账号）、角色分配、联系人建立。
- 确保员工 App 根据角色动态展示进销存模块（如仓库人员看不到价格，销售看不到成本等）。
- 支持批量邀请（可选）。

---

### 2. 用户故事

- **作为** 老板，我可以在“员工管理”页面点击“邀请员工”，选择角色（采购、销售、仓库、财务等），生成二维码或链接。员工扫码安装 ProClaw App 后，使用手机号注册，自动获得对应角色权限，并加入我的联系人列表。
- **作为** 新员工，收到邀请链接后，下载 App，输入手机号注册，登录后立即看到与自己角色匹配的工作界面（菜单、数据权限正确），并且老板会自动出现在我的联系人中。
- **作为** 已安装 App 但未分配角色的员工（例如从外部伙伴转为内部员工），老板也可以通过邀请重新分配角色。

---

### 3. 技术方案

#### 3.1 总体流程
1. **桌面端**（ProClaw Server）老板操作：
   - 进入“员工管理” → 点击“邀请员工” → 选择角色（可多选，支持一个员工多个角色） → 可选填写员工手机号（用于自动匹配） → 生成邀请码。
   - 邀请码存储于 `invitations` 表（复用 PRD v4.2 的表结构，新增 `role_ids` 字段）。
   - 生成二维码/链接，格式：`proclaw://invite?code=<邀请码>&host=<桌面端地址>&type=employee`
2. **员工** 扫码或点击链接：
   - 若未安装 App → 跳转下载页 → 安装后打开，解析邀请参数。
   - 若已安装 → 直接唤起 App，弹出提示：“您被邀请加入团队，是否接受？”
3. **移动端** 调用 `/api/invitations/accept_employee`，携带邀请码、员工手机号、设备信息。
4. **桌面端** 验证邀请码：
   - 检查是否有效、未过期、未使用。
   - 如果邀请时指定了手机号，则与请求中的手机号比对（不一致则拒绝）。
   - 创建用户（若不存在），类型为 `internal`，根据 `role_ids` 分配角色（插入 `user_roles`）。
   - 建立双向联系人关系（老板与员工之间，类型 `colleague`）。
   - 生成系统消息：“欢迎加入团队！你已被授予 XX 角色。”
   - 标记邀请码为 `used`。
5. **桌面端** 返回成功，并通过 WebSocket 推送：
   - 老板端：联系人列表更新，新员工上线。
   - 员工端：登录后立即获得正确的权限菜单。

#### 3.2 角色权限同步机制
- 桌面端 SQLite 中已有 `user_roles` 表。
- 移动端 App 在每次启动或切换角色时，调用 `/api/users/me` 获取当前用户的角色列表，并根据角色构建动态菜单（前端根据角色 code 显示/隐藏模块）。
- 权限校验在桌面端 API 层实施（已有中间件），移动端仅做 UI 隐藏，不依赖前端安全。

#### 3.3 新增/修改数据表

**扩展 `invitations` 表**（增加 `role_ids` 和 `invite_type` 字段）：
```sql
ALTER TABLE invitations ADD COLUMN invite_type TEXT DEFAULT 'external' CHECK(invite_type IN ('external', 'employee'));
ALTER TABLE invitations ADD COLUMN role_ids TEXT;  -- 存储角色ID的JSON数组，如 "[1,2]"
```

**复用 `user_contacts` 表**：员工间关系类型为 `colleague`。

#### 3.4 桌面端 API 扩展

**1. 创建员工邀请**  
`POST /api/invitations/create_employee`  
请求体：
```json
{
  "role_ids": [1, 3],           // 如采购+销售
  "target_phone": "13800138000"  // 可选，预填员工手机号
}
```
响应同外部邀请。

**2. 接受员工邀请**  
`POST /api/invitations/accept_employee`  
请求体：
```json
{
  "invite_code": "abc123",
  "phone": "13800138000",
  "name": "张三"
}
```
响应：
```json
{
  "success": true,
  "user_id": "user_002",
  "message": "您已成功加入团队，角色：采购、销售"
}
```

**3. 获取当前用户信息（含角色）**  
`GET /api/users/me`  
响应：
```json
{
  "id": "user_002",
  "name": "张三",
  "phone": "13800138000",
  "type": "internal",
  "roles": [
    { "id": 1, "name": "purchase" },
    { "id": 3, "name": "sales" }
  ],
  "permissions": ["create_purchase", "view_sales", ...]  // 可选，由后端计算
}
```

#### 3.5 移动端角色菜单动态实现

- 在 React Native 中维护一个路由配置表，每个模块关联所需的角色 code。
- 登录后获取用户角色，过滤可访问的模块，动态渲染底部 Tab 或侧边栏。
- 示例逻辑：
```js
const roleBasedRoutes = {
  'purchase': ['采购单', '供应商管理'],
  'sales': ['销售单', '客户管理'],
  'warehouse': ['库存管理', '入库/出库'],
  'finance': ['财务报表', '对账'],
  'boss': ['员工管理', '系统设置', ...所有模块]
};
```

#### 3.6 安全与防滥用
- 员工邀请同样设置有效期（建议7天）。
- 每个邀请码只能使用一次，使用后自动失效。
- 如果 `target_phone` 已指定，则只有该手机号能接受邀请；否则第一个使用该邀请码的手机号绑定。
- 邀请者权限检查：只有老板角色可以创建员工邀请（或拥有 `invite_employee` 权限）。

### 4. 界面设计要点

#### 4.1 桌面端（ProClaw 窗口）
- **员工管理页面**：列表展示现有员工，顶部按钮“邀请员工”。
- **邀请弹窗**：
  - 选择角色（多选复选框：采购、销售、仓库、财务）。
  - 可选填写员工手机号（非必填）。
  - 生成二维码 + 复制链接。
  - 提示：“员工扫码安装 App 后，使用该手机号注册即可自动加入团队。”

#### 4.2 移动端
- **注册/登录页**：若检测到邀请码（来自深度链接），在注册成功后自动调用接受邀请接口。
- **主界面**：根据角色动态显示菜单（例如仓库角色不显示“销售单”入口）。
- **系统消息**：收到“欢迎加入团队”消息，点击可查看自己的角色说明。

---

### 5. 与现有 PRD 的整合

- **复用 PRD v4.2 的邀请机制**，新增 `invite_type='employee'` 和 `role_ids` 字段。
- **复用现有权限中间件**：桌面端 API 根据 `user_roles` 校验权限，移动端仅做展示过滤。
- **员工联系人关系**：使用 `user_contacts` 表，类型 `colleague`，双向自动建立。

---

### 6. 实施步骤

1. **数据库迁移**：扩展 `invitations` 表，增加字段。
2. **桌面端 API**：
   - 实现 `create_employee` 邀请接口。
   - 实现 `accept_employee` 接口（处理角色分配、联系人建立）。
   - 扩展 `GET /users/me` 返回角色信息。
3. **桌面端 UI**：员工管理页面增加邀请功能。
4. **移动端**：
   - 深度链接解析支持 `type=employee`。
   - 注册/登录后调用 `accept_employee`。
   - 实现动态菜单渲染。
5. **测试**：
   - 老板邀请员工 → 员工扫码安装 → 注册 → 登录后菜单权限正确。
   - 员工手机号与邀请预填不匹配时拒绝。
   - 多个角色同时赋予，移动端菜单合并显示。
