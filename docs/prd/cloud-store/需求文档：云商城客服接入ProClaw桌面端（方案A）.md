## 1. 概述

### 1.1 背景

云商城（cloud-store）需要为商户提供客服功能，但根据架构决策：
- **云商城定位**：面向终端消费者的 B2C 商城系统，轻量化部署
- **避免重复建设**：客服功能已在 ProClaw 桌面端完整实现
- **统一体验**：商户已有 ProClaw 账号，在桌面端统一管理更便捷

### 1.2 决策

**方案A（采用）**：云商城前台不自己实现客服组件，访客点击"联系客服"时跳转到 ProClaw 桌面端的云客服页面统一处理。

### 1.3 目标

- 商户在 ProClaw 桌面端统一管理所有客服事务
- 云商城前台简化，不重复实现客服组件
- 访客体验流畅，客服入口清晰可见

## 2. 用户流程

### 2.1 访客（消费者）流程

```
访客访问云商城
    ↓
浏览商品/购物/查看订单
    ↓
点击"联系客服"入口
    ↓
【新窗口】打开 ProClaw 桌面端云客服页面
    ↓
与商户进行对话
```

### 2.2 商户（老板/运营）流程

```
商户登录 ProClaw 桌面端
    ↓
进入"云商城"后台（/shop）
    ↓
进入"客服管理"页面
    ↓
查看待回复消息、聊天记录、问答库
    ↓
回复客户问题
```

## 3. 技术方案

### 3.1 云商城侧修改

#### 3.1.1 添加客服入口

在以下页面添加"联系客服"入口：

| 页面 | 入口位置 | 行为 |
|------|---------|------|
| 商城首页 | 底部导航/页脚 | 跳转 |
| 商品详情页 | 商品信息下方 | 跳转 |
| 购物车页 | 结算按钮旁 | 跳转 |
| 订单详情页 | 订单状态区域 | 跳转 |

#### 3.1.2 跳转链接格式

```
https://proclaw.cc/customer-service?store={subdomain}
```

或通过登录态自动关联：
- 如果访客已登录，自动带上 `user_id`
- 如果未登录，引导先登录或使用匿名咨询

### 3.2 ProClaw 桌面端侧修改

#### 3.2.1 云客服页面改造

现有 `CustomerServicePage.tsx` 需要增强：

1. **租户识别**：从 URL 参数或登录态识别来自哪个商城
2. **商城筛选**：商户有多个商城时，可切换筛选
3. **商城信息展示**：显示当前咨询来自哪个商城

#### 3.2.2 路由参数

| 参数 | 说明 | 示例 |
|------|------|------|
| `store` | 商城子域名 | `my-shop` |
| `product` | 商品 ID（可选） | `prod_123` |
| `order` | 订单号（可选） | `DD20260609001` |

## 6. 数据流

### 6.1 客服消息存储

沿用现有 `customer_service_chat_logs` 表结构，新增字段：

```sql
ALTER TABLE customer_service_chat_logs 
ADD COLUMN store_subdomain VARCHAR(50);  -- 来源商城子域名
```

### 6.2 API 调用

```
云商城 → ProClaw API → 数据库
```

云商城调用 ProClaw API 记录咨询来源：
```
POST /api/customer-service/track
Body: {
  store_subdomain: "my-shop",
  page: "product_detail",
  product_id: "xxx",
  session_id: "xxx"
}
```

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 访客需要登录 ProClaw | 流失率高 | 提供匿名咨询表单兜底 |
| 跨域跳转体验 | 用户困惑 | 明确提示"将跳转到商户管理后台" |
| 多商城切换复杂 | 操作繁琐 | 优化筛选器，提供搜索 |
