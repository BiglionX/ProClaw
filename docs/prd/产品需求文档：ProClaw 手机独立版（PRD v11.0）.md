## 1. 背景与目标

### 1.1 背景
- 目前 ProClaw 手机端依赖于桌面端作为数据主节点，导致离线和独立使用受限。
- 大量用户（个体户、网约车司机）仅使用手机，需要完整功能且数据自主可控。
- 不同用户可能同时服务于多个公司（如一个司机为多个车队工作），需要身份隔离。

### 1.2 目标
- 手机端成为一等公民：拥有独立的本地 SQLite 数据库，所有业务操作优先本地。
- 数据同步由用户按需触发：支持云备份（端到端加密）和局域网直连两种方式，不强制依赖云。
- 插件按需下载：用户安装行业工作流插件时，才下载对应代码和数据库迁移。
- 多身份支持：一个手机 App 内可创建多个“身份”，每个身份的数据、插件、配置完全隔离。

## 2. 手机端独立数据库设计

### 2.1 整体架构

- 每个身份对应一个独立的 SQLite 数据库文件，存储路径：`{app_data}/profiles/{profile_id}/data.db`。
- 数据库包含所有业务表（商品、订单、客户、故事、收入等）以及系统表（同步元数据、插件注册表）。

### 2.2 核心数据表（每个身份）

沿用桌面端核心表结构，但增加以下字段以支持同步和多设备：

```sql
-- 全局同步元数据表
CREATE TABLE sync_metadata (
  id INTEGER PRIMARY KEY,
  last_sync_time INTEGER,          -- 上次同步时间戳
  device_id TEXT,                   -- 本设备唯一ID
  sync_token TEXT                   -- 用于增量同步的游标
);

-- 各业务表需增加 last_modified 和 sync_status
ALTER TABLE products ADD COLUMN last_modified INTEGER;
ALTER TABLE products ADD COLUMN sync_status TEXT DEFAULT 'clean'; -- clean/pending/conflict
```

### 2.3 变更日志（Change Log）

为支持增量同步，需维护一张变更日志表：

```sql
CREATE TABLE change_log (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  table_name TEXT,
  row_id TEXT,
  operation TEXT,   -- 'insert', 'update', 'delete'
  old_value JSON,
  new_value JSON,
  timestamp INTEGER,
  sync_status TEXT DEFAULT 'pending'  -- pending/synced
);
```

- 每次业务操作（CUD）自动写入 change_log。
- 同步完成后，将对应记录标记为 synced 或删除。

### 2.4 数据库加密（可选）

- 用户可设置数据库密码，使用 SQLCipher 对数据库文件进行 AES-256 加密。
- 密码仅存储在设备安全区域（Android Keystore / iOS Keychain），不同身份使用不同密钥。

## 3. 云备份的加密与同步协议（简版）

### 3.1 核心原则

- **端到端加密**：数据在手机端加密后上传，服务端无法解密。
- **用户完全控制**：加密密钥由用户密码派生，ProClaw 不存储密钥。
- **可选服务**：用户可选择是否开启云备份（默认关闭）。

### 3.2 加密方案

- 使用 AES-256-GCM 对每个数据块（如一条记录或一个文件）独立加密。
- 密钥派生：`key = PBKDF2(user_password, salt, iterations=100000, hash=SHA256)`。
- 每个数据块附带随机 IV 和认证标签。

### 3.3 同步协议（简化版）

**上传播放**（手机 → 云端）：
1. 客户端查询本地 change_log 中 `sync_status='pending'` 的记录。
2. 将记录按表分组，序列化为 JSON 后加密。
3. 上传到云存储（如 Supabase Storage），路径为 `/user/{user_id}/{device_id}/{table}/{timestamp}.enc`。
4. 上传成功后，将 change_log 状态更新为 `synced`，并记录 `last_sync_time`。

**下拉合并**（云端 → 手机）：
1. 客户端向云端请求 `last_sync_time` 之后其他设备上传的变更列表。
2. 下载加密文件，本地解密后解析为变更记录。
3. 应用冲突解决策略（见下节）合并到本地数据库。
4. 合并完成后更新 `last_sync_time`。

### 3.4 冲突解决策略

- **时间戳优先**：比较 `last_modified`，较新的覆盖。
- **字段级合并**：若手机修改了价格，电脑修改了库存，两者可合并。
- **手动处理**：冲突时在 UI 中展示差异，用户选择保留哪个版本（适用于关键数据如订单）。

### 3.5 用户密码管理

- 首次开启云备份时，要求用户设置“备份密码”（可不同于登录密码）。
- 备份密码丢失则无法恢复数据，App 内明确提示风险。
- 提供“导出恢复密钥”功能（打印二维码或保存文件）。

## 4. 局域网直连同步（可选）

### 4.1 适用场景

- 手机与电脑在同一 WiFi 下，用户希望不经过云端直接同步。
- 速度快、无流量费用、无隐私顾虑。

### 4.2 技术实现

- **设备发现**：使用 mDNS（Bonjour）广播服务 `_proclaw-sync._tcp.local.`，携带设备名称、身份ID、同步端口。
- **连接建立**：手机端主动扫描发现桌面端，发起 WebSocket 连接（或直接 HTTP/2）。
- **身份验证**：桌面端生成一个 6 位验证码，手机端输入完成配对（也可扫码）。
- **同步协议**：与云备份类似，但数据不加密传输（局域网信任），直接发送 change_log 的 JSON 体。
- **冲突处理**：与云端一致。

### 4.3 用户操作流程

1. 手机端点击“同步” → 选择“局域网同步”。
2. App 自动扫描同网络下的 ProClaw 桌面端，列出可用设备。
3. 用户选择目标设备，桌面端显示验证码。
4. 手机端输入验证码，建立安全通道。
5. 选择同步方向：“合并”（双向）、“仅发送至电脑”、“仅从电脑拉取”。
6. 开始同步，显示进度。

## 5. 插件按需安装机制

### 5.1 设计目标

- 手机端初始安装包体积小（不包含所有行业插件）。
- 用户仅在需要时下载特定插件（如餐厅工作流、维修工单）。
- 插件数据（表结构、前端代码）与当前身份绑定。

### 5.2 技术实现

- 插件信息从 FlowHub 获取，安装时下载 ZIP 包。
- 插件安装流程：
  1. 用户点击“安装” → 展示权限列表 → 确认。
  2. 下载 ZIP 到临时目录，校验签名。
  3. 解压到 `{profile_dir}/plugins/{plugin_id}/`。
  4. 执行数据库迁移脚本（`up.sql`），在身份数据库中创建插件所需的表。
  5. 动态加载前端组件（通过 `import()` 或 `eval`），注册路由和菜单。
  6. 如果是后端插件（Rust），由于手机端无法动态加载原生库，需使用 WebAssembly 或纯 JS 实现。因此手机端插件暂时只支持纯前端 + JS 逻辑，不能有原生依赖。

### 5.3 插件更新与卸载

- **更新**：检查新版本，下载替换，执行增量迁移脚本。
- **卸载**：执行 `down.sql` 删除表，删除插件目录，从菜单移除。

### 5.4 与 AI Team 的集成

- 插件可声明它依赖或推荐的 AI Team（如餐厅插件推荐“后厨优化 Agent”）。
- 用户在安装插件时，系统可提示“是否同时安装推荐的 AI Team？”。
- 安装后，AI Team 能读取插件创建的表数据（需授权）。

## 6. 多身份数据隔离

### 6.1 设计需求

- 一个用户可能同时服务于多个公司（个人店铺、兼职司机、自由职业者）。
- 每个身份的数据、插件、配置、同步目标完全独立，互不干扰。
- 切换身份时，App 界面、数据库、菜单均需重新加载。

### 6.2 实现方案

- **身份管理表**（全局，独立于身份数据库）：
  ```sql
  CREATE TABLE profiles (
    id TEXT PRIMARY KEY,
    name TEXT,
    avatar TEXT,
    db_path TEXT,           -- 数据库文件路径
    created_at INTEGER,
    last_used INTEGER
  );
  ```
- App 启动时加载最近使用的身份，或显示身份选择页。
- 切换身份时：
  1. 关闭当前数据库连接。
  2. 释放当前插件资源。
  3. 加载新身份的数据库，并重新初始化插件（已安装的插件列表从新身份数据库中读取）。
  4. 刷新 UI（菜单、当前页面）。

### 6.3 数据共享机制（可选）

- 允许用户将某个身份的数据“导出”到另一个身份（例如把个人身份的商品库复制到公司身份）。
- 提供“合并数据”向导，处理重复记录。

### 6.4 备份与同步的身份隔离

- 每个身份独立开启云备份，使用不同的加密密码（可复用同一密码但派生不同密钥）。
- 局域网同步时，只能与相同身份 ID 的设备配对（防止数据错乱）。

- [ ] 所有同步操作不丢失数据，冲突时按策略正确处理。
